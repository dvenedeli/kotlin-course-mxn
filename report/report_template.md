### ВВЕДЕНИЕ

**Актуальность.**  
Кратко описать, почему необходима система учёта нотариальных дел: ведение реестра дел, быстрый поиск, контроль дат открытия/закрытия, анализ сумм гонорара, снижение ошибок ручного учета.

**Цель работы.**  
Разработка консольного приложения на языке Kotlin для учёта, хранения и обработки данных нотариальных дел.

**Задачи работы.**
- выполнить анализ предметной области «нотариальные дела»;
- сформулировать функциональные и нефункциональные требования;
- спроектировать модель данных;
- реализовать консольное меню и функционал CRUD (создание/чтение/обновление/удаление);
- реализовать поиск, сортировку, фильтрацию;
- реализовать расчёт агрегированных показателей (среднее, сумма, минимум, максимум);
- реализовать загрузку/сохранение данных в JSON/CSV;
- выполнить тестирование приложения и зафиксировать результаты.

**Структура работы.**  
Работа состоит из введения, трёх глав основной части, заключения, списка литературы и приложений.

## ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

### 1.1 Описание предметной области
В рамках работы рассматривается учёт нотариальных дел. Под нотариальным делом понимается запись, содержащая идентификатор, номер дела, ФИО клиента, тип нотариального действия, дату открытия/закрытия и сумму гонорара.\n\
\n\
**Пользователь системы:** сотрудник (оператор), который ведёт список дел.\n\
\n\
### 1.2 Требования к системе\n\
\n\
**Функциональные требования:**\n\
- загрузка данных из файла JSON при запуске программы;\n\
- отображение всех дел;\n\
- добавление нового дела;\n\
- редактирование дела по идентификатору;\n\
- удаление дела;\n\
- поиск дел по текстовым полям (номер, ФИО, тип);\n\
- сортировка дел (по выбранному полю);\n\
- фильтрация дел по типу;\n\
- вычисление агрегированных показателей по сумме гонорара (average/sum/min/max);\n\
- сохранение данных в JSON или CSV;\n\
- корректная обработка некорректного ввода пользователя.\n\
\n\
**Нефункциональные требования:**\n\
- приложение консольное, с циклическим меню;\n\
- данные хранятся в локальных файлах `.json`/`.csv`;\n\
- устойчивость к ошибкам ввода (не падать при ошибочных значениях);\n\
- сборка и запуск на ОС Windows.\n\
\n\
### 1.3 Входные/выходные данные, ограничения\n\
\n\
**Входные данные:**\n\
- ввод пользователя в консоли;\n\
- файл данных JSON/CSV.\n\
\n\
**Выходные данные:**\n\
- отображение списка/результатов операций в консоли;\n\
- сохранённый файл JSON/CSV.\n\
\n\
**Ограничения:**\n\
- формат дат: `YYYY-MM-DD`;\n\
- CSV использует разделитель `,` и ожидает 7 полей: `id,caseNumber,clientName,caseType,openDate,closeDate,fee`.\n\
\n\

## ГЛАВА 2. РАЗРАБОТКА КОНСОЛЬНОГО ПРИЛОЖЕНИЯ\n\
\n\
### 2.1 Модель данных\n\
\n\
**Сущность NotaryCase** содержит поля:\n\
- `id` — идентификатор записи;\n\
- `caseNumber` — номер дела;\n\
- `clientName` — ФИО клиента;\n\
- `caseType` — тип нотариального действия;\n\
- `openDate` — дата открытия;\n\
- `closeDate` — дата закрытия (может отсутствовать);\n\
- `fee` — сумма гонорара.\n\
\n\
### 2.2 Пользовательский интерфейс (консольное меню)\n\
\n\
Приложение запускается с отображением циклического меню. Пользователь выбирает пункт меню, после чего вводит необходимые данные. Для чтения и проверки ввода используются функции `readInt`, `readDouble`, `readLocalDate`, `readNotEmptyString`.\n\
\n\
### 2.3 Работа с JSON/CSV-файлом\n\
\n\
При запуске выполняется загрузка из файла JSON (по умолчанию `notary_cases.json`). Пользователь может сохранить данные или загрузить данные из JSON/CSV через пункты меню.\n\
\n\
**Пример JSON (фрагмент):**\n\
```json\n\
[\n\
  {\n\
    \"id\": 1,\n\
    \"caseNumber\": \"2\",\n\
    \"clientName\": \"...\",\n\
    \"caseType\": \"...\",\n\
    \"openDate\": \"2024-01-01\",\n\
    \"closeDate\": null,\n\
    \"fee\": 500.0\n\
  }\n\
]\n\
```\n\
\n\
**Пример CSV (фрагмент):**\n\
```text\n\
id,caseNumber,clientName,caseType,openDate,closeDate,fee\n\
1,001,Иванов И.И.,Доверенность,2024-01-01,,500.0\n\
```\n\
\n\
### 2.4 Реализация функционала приложения\n\
\n\
**CRUD:** добавление, редактирование по `id`, удаление.\n\
\n\
**Поиск:** по номеру дела, ФИО клиента, типу действия.\n\
\n\
**Сортировка:** по одному из полей (ID/номер/ФИО/дата открытия/сумма).\n\
\n\
**Фильтрация:** по типу дела.\n\
\n\
**Агрегаты:** вычисление среднего, суммы, минимума и максимума по `fee`.\n\
\n\
**Структура проекта (кратко):**\n\
- `ui` — ввод и форматирование вывода;\n\
- `app` — прикладной слой (use-cases);\n\
- `domain` — логика работы со списком дел;\n\
- `data` — репозитории JSON/CSV.\n\
