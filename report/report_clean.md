

## ВВЕДЕНИЕ

**Актуальность.**  
Ведение учета нотариальных дел требует хранения структурированных сведений о каждом деле, быстрого поиска по реквизитам и контроля статуса (дело в работе или закрыто). При ручном учете в таблицах или блокнотах возрастает вероятность ошибок при вводе, дублирования записей и потери актуальности данных. Кроме того, для принятия управленческих решений необходимо оперативно получать агрегированные показатели (сумма, среднее, минимум и максимум гонорара по всем делам), что без автоматизации занимает значительное время.

Разработка специализированного программного средства позволяет централизовать хранение данных, стандартизировать формат записей и обеспечить единообразные операции поиска, сортировки и фильтрации. В рамках учебной дисциплины «Программирование на языке Kotlin» целесообразно реализовать консольное приложение, демонстрирующее применение объектно-ориентированного подхода, коллекций, сериализации и разделения ответственности между слоями программы.

**Объект исследования** — процесс учета и обработки данных о нотариальных делах.

**Предмет исследования** — методы и средства реализации консольного приложения на языке Kotlin для хранения и обработки таких данных с сохранением в файлы JSON и CSV.

**Цель работы.**  
Разработка консольного приложения на языке Kotlin для учета, хранения и обработки данных нотариальных дел.

## ГЛАВА 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ

### 1.1 Описание предметной области

Предметная область работы — учет нотариальных дел. В рамках разрабатываемой системы **нотариальное дело** — это запись (учетная карточка), описывающая одно нотариальное действие или пакет действий в отношении клиента. Каждая запись однозначно идентифицируется внутренним номером (ID) и содержит реквизиты, необходимые для учета и отчетности.

**Пользователь системы** — оператор (сотрудник), который:
- ведет реестр открытых и закрытых дел;
- добавляет новые дела при обращении клиентов;
- обновляет данные при изменении статуса или суммы гонорара;
- выполняет поиск по ФИО, номеру дела или типу действия;
- формирует сводные показатели по гонорарам.

**Жизненный цикл дела** в приложении можно описать следующими состояниями:
1. **Создание** — ввод реквизитов, присвоение ID, дата открытия фиксируется (или вводится пользователем).
2. **В работе** — дело присутствует в списке, поле даты закрытия пустое (`closeDate = null`).
3. **Закрыто** — указана дата закрытия, дело остается в архиве для поиска и отчетов.
4. **Удаление** — запись удаляется из текущего списка в памяти (при сохранении в файл изменения фиксируются).

Типовой сценарий работы оператора:
1. Запуск приложения и автоматическая загрузка данных из файла.
2. Просмотр списка дел или поиск нужной записи.
3. Добавление нового дела или редактирование существующего.
4. При необходимости — сортировка, фильтрация, просмотр статистики по гонорарам.
5. Сохранение актуального состояния в JSON или CSV перед завершением сеанса.

**Таблица 1.1 — Атрибуты нотариального дела в системе**

| Поле | Тип (логический) | Назначение | Пример |
|------|------------------|------------|--------|
| id | целое число | Уникальный идентификатор записи | 1 |
| caseNumber | строка | Номер дела по внутреннему учету | «12/2024» |
| clientName | строка | ФИО клиента | «Иванов И.И.» |
| caseType | строка | Вид нотариального действия | «Доверенность» |
| openDate | дата | Дата открытия дела | 2024-01-15 |
| closeDate | дата или пусто | Дата закрытия; пусто — «в работе» | 2024-02-01 или — |
| fee | вещественное число | Сумма гонорара, руб. | 3500.00 |

Основная сущность — **NotaryCase** (нотариальное дело), реализованная как `data class` в пакете `org.example.models`. Использование `data class` оправдано необходимостью хранения набора полей, операций копирования при редактировании (`copy`) и сериализации в JSON.

**Таблица 2.1 — Поля модели NotaryCase**

| Поле Kotlin | Тип Kotlin | Обязательность | Примечание |

| clientName | String | да | Не пустая строка |

### 2.3 Работа с JSON/CSV-файлом

Данные между сеансами хранятся в файлах. При старте вызывается загрузка `notary_cases.json`. Сохранение и дополнительная загрузка — через меню.

**Алгоритм загрузки данных из файла:**
1. Пользователь указывает путь и формат (или используется путь по умолчанию при старте).
2. Создается реализация `NotaryCaseRepository` (JSON или CSV).
3. Вызывается `load()`: чтение файла, разбор содержимого.
4. При успехе список в `NotaryCaseStore` заменяется загруженными записями, пересчитывается `nextId`.
5. При ошибке возвращается `Result.failure`, в UI выводится сообщение.

**Алгоритм сохранения:**
1. Получить копию списка дел из хранилища.
2. Передать список в `repository.save(cases)`.
3. Для JSON — сериализация всего списка с `prettyPrint`.
4. Для CSV — формирование строки с заголовком и строками данных через `CsvNotaryCaseCodec`.
5. Запись в файл; при успехе — сообщение пользователю.

**Пример JSON:**

```json
[
  {
    "id": 1,
    "caseNumber": "001",
    "clientName": "Иванов И.И.",
    "caseType": "Доверенность",
    "openDate": "2024-01-01",
    "closeDate": null,
    "fee": 500.0
  }
]
```

**Пример CSV:**

```text
id,caseNumber,clientName,caseType,openDate,closeDate,fee
1,001,Иванов И.И.,Доверенность,2024-01-01,,500.0
```

**Таблица 2.4 — Сравнение форматов JSON и CSV**

| Критерий | JSON | CSV |
|----------|------|-----|
| Структура | Иерархический массив объектов | Таблица, первая строка — заголовок |
| Читаемость для человека | Высокая (с отступами) | Средняя |
| Поддержка null для closeDate | Да (`null`) | Пустое поле |
| Удобство правки в Excel | Низкое | Высокое |
| Реализация в проекте | kotlinx.serialization | Собственный кодек |

### 2.4 Реализация функционала приложения

Программа организована по **слоям**:

| Слой | Пакет / класс | Ответственность |
|------|---------------|-----------------|
| UI | `Main.kt`, `ConsoleUI`, `NotaryCaseFormatter` | Меню, ввод, вывод строк пользователю |
| App | `NotaryCaseApp` | Сценарии: связь меню с доменом и файлами |
| Domain | `NotaryCaseStore` | CRUD, поиск, сортировка, фильтр, статистика |
| Data | `JsonNotaryCaseRepository`, `CsvNotaryCaseRepository`, `CsvNotaryCaseCodec` | Чтение/запись файлов |

**Рисунок 2.3** — схема слоев (блоки: UI → App → Domain; App → Data → Files).

**Операция «Добавление»:** пользователь вводит поля; создается `NotaryCase` с `id = 0`; `NotaryCaseStore.addCase` присваивает новый ID и добавляет в `MutableList`.

**Операция «Редактирование»:** поиск по ID; пошаговый ввод новых значений (пустой ввод — оставить старое); `updateCase`.

**Операция «Удаление»:** `deleteCase(id)` — удаление из списка по предикату.

**Поиск:** регистронезависимое вхождение подстроки в `caseNumber`, `clientName`, `caseType`.

**Сортировка:** выбор поля (`SortField`: ID, номер, ФИО, дата открытия, гонорар) и направления (возрастание/убывание); `sortedWith`.

**Фильтрация:** вхождение подстроки типа дела в `caseType` без учета регистра.

**Статистика:** по полю `fee` вычисляются среднее арифметическое, сумма, минимум и максимум; для пустого списка — нули.

**Фрагмент логики хранения (пояснение к реализации):**  
Список дел хранится в оперативной памяти в `NotaryCaseStore`; файлы используются для персистентности между запусками. Такой подход соответствует требованиям учебного консольного приложения и упрощает unit-тестирование доменного слоя без файловой системы.
